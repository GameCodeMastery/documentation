The `BP_GameplayEffect` class in the Advanced Abilities System, designed for Unreal Engine 5, is a Blueprint-based class that manages modifications to attributes within the Advanced Attributes System (e.g., `Attribute.Health`, `Attribute.Stamina`). Its purpose is to provide a data-driven framework for applying instant, temporary, or persistent changes to gameplay attributes, such as dealing damage, granting buffs, or applying debuffs like poison, enhancing dynamic gameplay in Action RPGs.

This compound data type addresses the need for flexible, designer-friendly attribute manipulation without requiring extensive coding. `BP_GameplayEffect` is a standalone class that directly interacts with the Advanced Attributes System to modify base attribute values, offering a robust solution for stat-driven mechanics like health reduction or stat boosts.

---

## Basic Usage

Gameplay Effects are applied via the `BP_AdvancedAbilitySystemComponent` to modify attributes, trigger Gameplay Cues, or grant abilities. They are typically used within Gameplay Abilities or other systems to enact gameplay changes. Below are the key functions, their purposes, and how they can be used in Blueprints.

1. **Apply Effect**
    - **Purpose**: Executes the logic when the Gameplay Effect is applied, such as modifying attributes or spawning cues.
    - **Usage**:
        - Override in a child Blueprint to add custom functionality (e.g., triggering additional effects).
        - Example: In `BP_DamageEffect`, reduce health and spawn a blood particle.
            ```blueprint
            Apply Effect -> Call Parent Function -> Modify Attribute (AttributeTag: Attribute.Health, Value: -25) -> Play Gameplay Cue (Cue: BP_BloodImpact)
            ```

2. **Remove Effect**
    - **Purpose**: Handles cleanup when the Gameplay Effect is removed, such as reverting temporary attribute changes.
    - **Usage**:
        - Override to add custom cleanup logic (e.g., stopping visual effects).
        - Example: In `BP_SpeedBuff`, remove a speed boost.

            ```blueprint
            Remove Effect -> Call Parent Function -> Stop Emitter at Location (Emitter: SpeedParticles)
            ```

3. **Can Apply Effect**
    - **Purpose**: Checks whether the Gameplay Effect can be applied based on conditions like Gameplay Tags.
    - **Usage**:
        - Override to enforce custom conditions (e.g., requiring specific actor states).
        - Example: In `BP_HealEffect`, check if the target has health below maximum.
            ```blueprint
            Can Apply Effect -> Get Current Attribute Value (AttributeTag: Attribute.Health) -> Branch (Condition: Health < MaxHealth) -> True: Return True
            ```

## Key Properties

|Property Name|Purpose|
|---|---|
|`Duration`|Defines the effect’s lifespan. Set to 0 for instant effects (e.g., damage), a positive value for temporary effects (e.g., 10 seconds for a buff), or -1 for persistent effects.|
|`Modifiers`|An array of attribute modifiers that define how the effect alters attributes (e.g., reduce `Attribute.Health` by 25). Modifier IDs are auto-generated by the system.|
|`Executions`|An array of execution classes (child of `BP_Execution`) that run custom logic when the effect is applied, such as complex damage calculations.|
|`Application Required Tags`|Gameplay Tags the owning actor must have for the effect to be applied (e.g., `State.Alive`).|
|`Application Blocked Tags`|Gameplay Tags that prevent the effect from being applied if present on the owning actor (e.g., `State.Immune`).|
|`Period`|The interval (in seconds) for applying ongoing effects (e.g., 1 for poison damage every second).|
|`Granted Abilities`|An array of `BP_AdvancedGameplayAbility` classes granted when the effect is applied (e.g., `BP_SprintAbility` for a speed buff).|
|`Stacking`|Defines the policy for applying the effect multiple times (e.g., add stacks to increase intensity). Includes stack count and behavior settings.|
|`Gameplay Cues`|An array of `BP_GameplayCue` or `BP_GameplayCueActor` assets triggered when the effect is applied (e.g., `BP_BloodImpact` for damage).|

## Key Concepts

### Attribute Modification

`BP_GameplayEffect` is the primary mechanism for modifying attributes in the Advanced Attributes System, enabling instant, temporary, or persistent changes. It interacts directly with attributes (e.g., `Attribute.Health`) without relying on the Extended Attributes System, ensuring straightforward integration with the `BP_AdvancedAbilitySystemComponent`.

- **Implementation**: Configure `Modifiers` in the Details panel to define attribute changes (e.g., reduce `Attribute.Health` by 25). Apply via `BP_AdvancedAbilitySystemComponent`:
    
    ```blueprint
    ActivateAbility -> Get BP_AdvancedAbilitySystemComponent -> Apply Gameplay Effect By Class (Class: BP_DamageEffect)
    ```
    
- **Use Cases**: Instant damage (e.g., weapon hits), temporary buffs (e.g., speed boost for 10 seconds), or persistent effects (e.g., poison dealing damage over time).
- **Best Practices**: Use `Can Apply Effect` to enforce conditions (e.g., preventing healing on dead actors). Test attribute changes with `GetCurrentAttributeValue` to verify effects (see [[Attribute Progress Bar Widget]]).

### Data-Driven Design

Gameplay Effects are designed to be data-driven, allowing designers to create and customize effects by modifying properties in the Details panel without extensive Blueprint scripting. This makes it easy to balance gameplay mechanics like damage, buffs, or debuffs.

- **How It Works**: Properties like `Duration`, `Modifiers`, and `Gameplay Cues` define the effect’s behavior. For example, a `BP_SpeedBuff` might set `Duration` to 30 seconds, add a `Modifiers` entry to increase `Attribute.MoveSpeed` by 50%, and include a `Gameplay Cues` entry for `BP_SpeedParticleCue`.
- **Customization**: Create child Blueprints (e.g., `BP_PoisonEffect`) and adjust properties:
    
    ```blueprint
    // Details Panel: Duration: -1, Period: 1, Modifiers: Reduce Attribute.Health by 5, Gameplay Cues: BP_PoisonCue
    ```
    
- **Use Cases**: Create varied effects like health regeneration, stat debuffs, or item-based buffs without coding.
- **Best Practices**: Start with simple modifiers for testing, then add `Executions` for complex logic. Avoid modifying `Modifiers` IDs, as they are auto-generated.

### Stacking and Tag-Based Conditions

The `Stacking` property allows Gameplay Effects to be applied multiple times, increasing their intensity (e.g., stacking poison for higher damage). Tag-based conditions (`Application Required Tags`, `Application Blocked Tags`) ensure effects are applied only under specific circumstances, enhancing gameplay precision.

- **Stacking**:
    - Configure `Stacking` to define how multiple applications interact (e.g., add stacks up to a limit). Example: A `BP_PoisonEffect` with `Stacking` set to allow 3 stacks, each increasing damage.
    - Example:

        ```blueprint
        Apply Gameplay Effect By Class (Class: BP_PoisonEffect) -> Check Stack Count -> Print String (Current Stacks)
        ```

- **Tag-Based Conditions**:
    - Set `Application Required Tags` (e.g., `State.Alive`) and `Application Blocked Tags` (e.g., `State.Immune`) to control effect eligibility.
    - Example: Prevent a `BP_HealEffect` from applying to immune actors:

        ```blueprint
        Can Apply Effect -> Check Actor Has Tag (Tag: State.Immune) -> Branch (Condition: False) -> True: Return True
        ```

- **Use Cases**: Stackable debuffs (e.g., bleeding), conditional buffs (e.g., only apply to allies), or limited-use effects (e.g., one-time shields).
- **Best Practices**: Test stacking behavior with multiple applications to ensure balance. Use consistent Gameplay Tags across abilities and effects to avoid mismatches.
